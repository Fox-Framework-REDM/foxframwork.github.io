## Origin

`fox_banking` is converted from the original `ox_banking` resource by Overextended.

The Fox version keeps the banking feature set but adapts runtime integration for RedM and `fox_core`.

## Runtime Flow

- The client opens the NUI with `exports.fox_banking:openBank(bankId?)`.
- The NUI sends banking intent to the client.
- The client calls server callbacks through `ox_lib`.
- The server validates account access, branch context, amount, permissions, inventory state, and vault access before changing data.
- The server updates accounts, logs, pending transfers, invoices, or vault authorization.

## RedM Adaptation

- Bank access is built around RedM branch coordinates and bank clerk peds instead of GTA-style ATMs.
- `fox_interact` is the default interaction bridge for clerk peds.
- `ox_target` zones can be enabled with `UseOxTarget`.
- RedM bank door states are applied from `data/config.json` when `ApplyBankDoors` is enabled.

## Data Ownership

- `fox_core` owns player and account identity.
- `fox_banking` owns branch metadata, pending transfers, vault rent records, banking logs, and banking UI flow.
- `ox_inventory` owns carried money, gold, and vault inventory storage.
- `oxmysql` owns all persistent banking rows through parameterized queries.

## Security Model

- Client and NUI only request intent.
- Server callbacks validate the account, character, access level, amount, branch, and operation before applying changes.
- Vault access is authorized server-side and limited by account permission, configured distance, cooldown, and inventory hooks.
- Delayed transfers are processed on the server by a timed worker instead of trusting the client.

## Performance Notes

- Branches and config are static data files.
- Pending transfers are processed on a configured interval with `transferTickSeconds`.
- Completed delayed transfer rows are retained only for `completedTransferRetentionDays`.
- Bank doors are applied with a limited retry configuration instead of a tight loop.
